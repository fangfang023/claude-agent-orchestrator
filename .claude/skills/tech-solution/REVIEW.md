# 技术方案质量审查标准

## 何时使用

**触发条件：**
- 用户要求检查技术方案质量
- 技术方案生成完成后的自动审查
- 方案提交前的最终检查

## 审查流程

### 第一步：结构完整性检查

**必需模块检查：**

| 模块 | 必需 | 检查要点 |
|------|------|----------|
| 文档目的 | ✅ | 用途、受众、范围是否明确 |
| 核心技术概述 | ✅ | 是否提炼了关键技术点 |
| 详细操作步骤 | ✅ | 步骤是否清晰可执行 |
| 常见问题解决 | ✅ | 是否至少3项 |

**结构评分：**
- ⭐⭐⭐⭐⭐ 4个模块完整且详细
- ⭐⭐⭐⭐ 4个模块完整但不够详细
- ⭐⭐⭐ 缺少次要内容
- ⭐⭐ 缺少重要模块
- ⭐ 缺少核心模块

### 第二步：内容质量检查

#### 2.1 文档目的

**检查清单：**
- [ ] 用途说明清晰明确
- [ ] 目标受众定位准确
- [ ] 适用范围界定合理
- [ ] 与实际需求匹配

**常见问题：**
- ❌ 用途描述模糊或过于宽泛
- ❌ 受众定位不清晰
- ❌ 缺少适用范围说明

#### 2.2 核心技术概述

**检查清单：**
- [ ] 技术背景描述准确
- [ ] 创新点提炼精准（1-3个）
- [ ] 技术架构清晰
- [ ] 关键参数具体

**创新点评估：**
- ⭐⭐⭐⭐⭐ 创新点明确且有突破性
- ⭐⭐⭐⭐ 创新点清晰
- ⭐⭐⭐ 有一定创新
- ⭐⭐ 创新点不明显
- ⭐ 无创新点

**常见问题：**
- ❌ 创新点过多或过少
- ❌ 技术原理描述不清楚
- ❌ 缺少关键技术参数

#### 2.3 详细操作步骤

**检查清单：**
- [ ] 总体流程合理
- [ ] 步骤分解恰当
- [ ] 每个步骤包含：
  - 目标
  - 输入
  - 操作内容
  - 输出
  - 验证标准
- [ ] 步骤之间逻辑连贯

**可实施性评估：**
- ⭐⭐⭐⭐⭐ 步骤清晰，可直接实施
- ⭐⭐⭐⭐ 步骤较清晰，稍作调整可实施
- ⭐⭐⭐ 步骤基本清晰，需要补充细节
- ⭐⭐ 步骤不够清晰，难以实施
- ⭐ 步骤混乱，无法实施

**常见问题：**
- ❌ 步骤过于笼统
- ❌ 缺少必要的细节
- ❌ 步骤顺序不合理
- ❌ 缺少输入输出说明

#### 2.4 常见问题与解决

**检查清单：**
- [ ] 至少包含3个问题
- [ ] 问题描述具体
- [ ] 原因分析合理
- [ ] 解决方案可行
- [ ] 预防措施有效

**常见问题：**
- ❌ 问题数量不足（少于3个）
- ❌ 问题描述模糊
- ❌ 缺少原因分析
- ❌ 解决方案不具体

### 第三步：专业性和可读性检查

#### 3.1 语言专业性

**检查清单：**
- [ ] 使用标准技术术语
- [ ] 避免口语化表达
- [ ] 语言简洁准确
- [ ] 无语法错误
- [ ] 术语前后一致

**语言评分：**
- ⭐⭐⭐⭐⭐ 专业规范，无瑕疵
- ⭐⭐⭐⭐ 基本专业，个别可优化
- ⭐⭐⭐ 有一定专业性，需改进
- ⭐⭐ 专业性不足
- ⭐ 缺乏专业性

**常见问题：**
- ❌ 使用口语化表达
- ❌ 术语使用不一致
- ❌ 语言冗余啰嗦
- ❌ 存在语法错误

#### 3.2 逻辑严谨性

**检查清单：**
- [ ] 整体逻辑清晰
- [ ] 章节之间衔接自然
- [ ] 论述有据可依
- [ ] 无逻辑矛盾
- [ ] 推理过程合理

**常见问题：**
- ❌ 章节之间缺乏衔接
- ❌ 论述缺乏依据
- ❌ 存在逻辑矛盾

#### 3.3 结构可读性

**检查清单：**
- [ ] 层次分明
- [ ] 标题级别正确
- [ ] 编号统一规范
- [ ] 适当使用图表
- [ ] 版式整洁美观

**常见问题：**
- ❌ 层次混乱
- ❌ 标题级别错误
- ❌ 编号不统一
- ❌ 缺少必要的图表

### 第四步：可实施性检查

**关键评估点：**

| 评估维度 | 检查内容 | 权重 |
|----------|----------|------|
| 技术可行性 | 方案在技术上是否可实现 | 30% |
| 资源可获得性 | 所需资源是否可获得 | 20% |
| 时间合理性 | 实施周期是否合理 | 15% |
| 成本可控性 | 成本是否在可接受范围 | 15% |
| 风险可控性 | 风险是否可控 | 20% |

**可实施性评分：**
- ⭐⭐⭐⭐⭐ 高度可行，建议立即实施
- ⭐⭐⭐⭐ 可行，建议小规模试点
- ⭐⭐⭐ 基本可行，需要优化方案
- ⭐⭐ 可行性存疑，需要重大调整
- ⭐ 不可行，需要重新设计

## 综合评分

### 评分计算

```
总分 = 结构完整性(20%) + 内容质量(40%) + 专业性(20%) + 可实施性(20%)

各维度得分转换为百分制后加权计算
```

### 评级标准

| 等级 | 分数范围 | 建议 |
|------|----------|------|
| 优秀 | 90-100 | 可以直接使用 |
| 良好 | 80-89 | 稍作优化后使用 |
| 合格 | 70-79 | 需要改进后使用 |
| 待改进 | 60-69 | 需要重大修改 |
| 不合格 | 0-59 | 需要重新撰写 |

## 常见质量问题速查

### 高频问题 TOP 10

| 排名 | 问题 | 影响 | 修改建议 |
|------|------|------|----------|
| 1 | 操作步骤过于笼统 | 严重影响可实施性 | 细化每个步骤，添加具体操作 |
| 2 | 创新点提炼不准确 | 影响技术价值 | 重新分析技术特点，精准提炼 |
| 3 | 缺少必要的图表 | 影响理解 | 添加架构图、流程图等 |
| 4 | 术语使用不一致 | 影响专业性 | 统一术语，添加术语表 |
| 5 | 常见问题少于3个 | 不符合标准 | 补充到至少3个 |
| 6 | 缺少验证标准 | 影响实施 | 为每个步骤添加验证标准 |
| 7 | 技术参数不具体 | 影响可实施性 | 添加具体的技术参数 |
| 8 | 逻辑不连贯 | 影响可读性 | 调整结构，增强衔接 |
| 9 | 语言口语化 | 影响专业性 | 改用专业术语 |
| 10 | 缺少预防措施 | 影响实用性 | 为每个问题添加预防措施 |

## 工作流程与质量保证

### 质量保证流程

**触发时机：**
- 每次技术方案生成完成后自动触发
- 与 `GENERATE.md` 配合使用

**工作流程：**
```
┌─────────────────────────────────────┐
│  步骤1: 生成技术方案 (GENERATE.md)   │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  步骤2: 自动质量审查 (本文件标准)     │
│  - 结构完整性检查                    │
│  - 内容质量检查                      │
│  - 专业性检查                        │
│  - 可实施性检查                      │
│  - 生成评分和评级                    │
└──────────────┬──────────────────────┘
               ↓
         综合评分 >= 70分？
         ├─ 是 → 继续后续工作（生成商业报告等）
         └─ 否 → 自动改正（最多3次）
                  ↓
        重新生成技术方案
                  ↓
         重新质量审查
                  ↓
         改正次数 < 3次？
         ├─ 是 → 继续审查
         └─ 否 → 强制通过（使用第3次版本）
                    ↓
              记录质量日志
                    ↓
              继续后续工作
```

### 改正机制

**改正触发条件：**
- 综合评分 < 70分（合格线）
- 或存在"必须修改"级别的问题

**改正次数限制：**
- 最多3次自动改正
- 每次改正后重新评分
- 第3次改正后无论评分如何，强制通过

**改正内容：**
1. **必须修改问题**（影响使用）：优先修正
   - 缺少核心模块
   - 创新点提炼不准确
   - 操作步骤过于笼统
   - 技术参数不具体

2. **建议优化内容**（提升质量）：时间允许则修正
   - 添加必要的图表
   - 完善技术细节
   - 优化语言表达

### 与 GENERATE.md 的配合

**职责分工：**
- `GENERATE.md`：定义"如何生成"技术方案
- `REVIEW.md`：定义"如何检查质量"和"如何改正"
- 两者配合形成完整的质量保证体系

**调用示例：**
```
1. 读取 GENERATE.md 中的生成步骤
2. 按照步骤生成技术方案初稿
3. 读取 REVIEW.md 中的审查标准
4. 按照标准审查技术方案
5. 根据审查结果决定是否需要改正
6. 重复步骤2-5，最多3次
7. 质量合格或达到3次上限后，输出最终版本
```

- [ ] 操作步骤有输入输出
- [ ] 常见问题至少3个

### 特殊情况处理

**信息不足场景：**
```
⚠️ 技术方案信息不足

当前创意描述不足以生成完整的技术方案。

缺少以下关键信息：
- [ ] 具体的技术实现方式
- [ ] 关键技术参数
- [ ] 与现有技术的差异
- [ ] 实施环境和约束

建议：
1. 提供更详细的技术描述
2. 说明核心创新点
3. 明确应用场景

选择：
1. 基于现有信息生成框架（会有[待补充]标注）
2. 等待补充信息后生成
```

**技术不可行场景：**
```
❌ 技术方案评估结果

经评估，该技术方案存在以下问题：
- 技术实现难度过高
- 成本超出合理范围
- 存在不可克服的技术障碍

建议：
1. 调整技术方案
2. 降低技术复杂度
3. 分阶段实施

是否调整技术方案？(Y/n)
```
