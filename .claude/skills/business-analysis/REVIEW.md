# 商业价值评估报告质量审查标准

## 何时使用

**触发条件：**
- 用户要求检查商业分析报告质量
- 商业分析报告生成完成后的自动审查
- 报告提交给管理层前的最终检查

## 审查流程

### 第一步：结构完整性检查

**必需章节检查：**

| 章节 | 必需 | 关键内容 | 检查要点 |
|------|------|----------|----------|
| 报告摘要 | ✅ | 核心结论、价值点、风险、行动 | 四要素齐全且明确 |
| 技术剖析 | ✅ | 匹配度、创新点、先进性 | 分析深度和准确性 |
| 商业价值 | ✅ | 市场、竞争、盈利模式 | 数据支撑和逻辑性 |
| 专利布局 | ✅ | 核心专利、外围布局、地域 | 策略完整性和可操作性 |
| 风险评估 | ✅ | 可专利性、FTO、泄露风险 | 风险识别和应对 |
| 行动方案 | ✅ | 综合结论、行动清单 | 具体性和时间性 |

**结构评分：**
- ⭐⭐⭐⭐⭐ 6个章节完整且深入
- ⭐⭐⭐⭐ 6个章节完整但深度一般
- ⭐⭐⭐ 主要章节完整
- ⭐⭐ 缺少重要章节
- ⭐ 缺少核心章节

### 第二步：内容质量检查

#### 2.1 报告摘要 (Executive Summary)

**检查清单：**
- [ ] 核心结论明确（优先申请/布局后申请/暂缓/不建议）
- [ ] 关键价值点一句话概括
- [ ] 核心风险具体明确
- [ ] 行动建议1-3条，具体可执行

**质量标准：**
- ⭐⭐⭐⭐⭐ 结论明确，依据充分，建议具体
- ⭐⭐⭐⭐ 结论明确，建议较具体
- ⭐⭐⭐ 结论较明确，建议不够具体
- ⭐⭐ 结论模糊
- ⭐ 无明确结论

**常见问题：**
- ❌ 核心结论不明确
- ❌ 价值点不够具体
- ❌ 风险提示缺失
- ❌ 行动建议空泛

#### 2.2 技术核心与创新点剖析

**检查清单：**
- [ ] 问题与方案匹配度评估
- [ ] 核心创新点提炼（1-3个）
- [ ] 技术先进性评估（水平+类型）
- [ ] 技术壁垒评估

**创新点质量：**
- ⭐⭐⭐⭐⭐ 创新点精准且有价值
- ⭐⭐⭐⭐ 创新点清晰
- ⭐⭐⭐ 有创新点但不够精准
- ⭐⭐ 创新点模糊
- ⭐ 无明确创新点

**常见问题：**
- ❌ 创新点数量不当（过多/过少）
- ❌ 技术特征描述不准确
- ❌ 先进性评估缺乏依据
- ❌ 壁垒分析不到位

#### 2.3 商业价值评估

**检查清单：**
- [ ] 市场规模分析（TAM/SAM/SOM）
- [ ] 应用场景和产品关联
- [ ] 竞争优势对比分析
- [ ] 商业化路径（4种模式）
- [ ] 战略契合度评估

**市场分析质量：**
- ⭐⭐⭐⭐⭐ 数据翔实，分析深入
- ⭐⭐⭐⭐ 有数据支撑，分析较深入
- ⭐⭐⭐ 有基本分析，数据不够充分
- ⭐⭐ 分析浅显
- ⭐ 无有效分析

**常见问题：**
- ❌ 缺少市场层级分析
- ❌ 竞争对比不具体
- ❌ 缺少量化数据
- ❌ 商业模式不清晰
- ❌ 战略契合度分析缺失

#### 2.4 专利布局策略

**检查清单：**
- [ ] 核心专利申请策略
- [ ] 权利要求布局方向
- [ ] 外围/防御性专利布局
- [ ] 地域布局策略（含表格）
- [ ] 申请时机和保密策略

**策略质量：**
- ⭐⭐⭐⭐⭐ 策略完整、具体、可操作
- ⭐⭐⭐⭐ 策略较完整、较具体
- ⭐⭐⭐ 有基本策略，不够具体
- ⭐⭐ 策略模糊
- ⭐ 无有效策略

**常见问题：**
- ❌ 权利要求布局缺失
- ❌ 外围布局考虑不足
- ❌ 地域布局缺少表格
- ❌ 申请时机分析不深入
- ❌ 缺少技术秘密策略

#### 2.5 风险评估与规避

**检查清单：**
- [ ] 可专利性风险（新颖性、创造性、实用性）
- [ ] FTO风险预警
- [ ] 技术泄露风险
- [ ] 应对措施具体

**风险识别质量：**
- ⭐⭐⭐⭐⭐ 风险识别全面，应对有效
- ⭐⭐⭐⭐ 风险识别较全面
- ⭐⭐⭐ 有基本风险识别
- ⭐⭐ 风险识别不足
- ⭐ 无有效风险识别

**常见问题：**
- ❌ 可专利性三性评估缺失
- ❌ FTO风险分析缺失
- ❌ 风险等级不明确
- ❌ 应对措施空泛

#### 2.6 结论与行动方案

**检查清单：**
- [ ] 综合结论明确
- [ ] 行动清单分时间阶段
- [ ] 每个行动具体可执行
- [ ] 有明确的时间节点

**行动方案质量：**
- ⭐⭐⭐⭐⭐ 行动具体、分期清晰、可执行
- ⭐⭐⭐⭐ 行动较具体、分期较清晰
- ⭐⭐⭐ 有行动方案但不够具体
- ⭐⭐ 行动方案模糊
- ⭐ 无有效行动方案

**常见问题：**
- ❌ 结论不明确
- ❌ 行动缺少时间分期
- ❌ 行动不够具体
- ❌ 缺少可执行性

### 第三步：专业性和逻辑性检查

#### 3.1 角色定位

**检查：**
- [ ] 体现IP总监的专业视角
- [ ] 从企业战略出发分析
- [ ] 具备技术、市场、法律综合视角
- [ ] 语言专业、简洁

**常见问题：**
- ❌ 视角单一（纯技术或纯市场）
- ❌ 缺乏战略高度
- ❌ 语言不专业

#### 3.2 逻辑严密性

**检查：**
- [ ] 分析结论有充分依据
- [ ] 章节之间逻辑连贯
- [ ] 数据和分析对应
- [ ] 无逻辑矛盾

**常见问题：**
- ❌ 结论缺乏依据
- ❌ 数据和结论不匹配
- ❌ 存在逻辑矛盾

#### 3.3 数据准确性

**检查：**
- [ ] 技术描述准确
- [ ] 市场数据合理
- [ ] 对比分析客观
- [ ] 无明显错误

**常见问题：**
- ❌ 技术理解偏差
- ❌ 市场数据夸大
- ❌ 对比分析不客观

### 第四步：实用性和可操作性检查

#### 4.1 决策参考价值

**评估维度：**
| 维度 | 评估内容 |
|------|----------|
| 结论明确性 | 是否给出明确的申请建议 |
| 依据充分性 | 结论是否有充分分析支撑 |
| 风险揭示 | 是否揭示主要风险 |
| 可操作性 | 行动方案是否可执行 |

**评分：**
- ⭐⭐⭐⭐⭐ 高决策参考价值
- ⭐⭐⭐⭐ 较高决策参考价值
- ⭐⭐⭐ 有一定参考价值
- ⭐⭐ 参考价值有限
- ⭐ 无参考价值

#### 4.2 管理层友好度

**检查：**
- [ ] 摘要清晰，便于快速理解
- [ ] 详细分析在后续章节
- [ ] 有明确的行动建议
- [ ] 时间节点清晰

**常见问题：**
- ❌ 摘要过长或不清晰
- ❌ 缺少明确的行动建议
- ❌ 时间节点不明确

## 综合评分

### 评分计算

```
总分 = 结构完整性(15%) + 内容质量(40%) + 专业性(20%) + 实用性(25%)
```

### 评级标准

| 等级 | 分数范围 | 建议 |
|------|----------|------|
| 优秀 | 90-100 | 可直接提交管理层 |
| 良好 | 80-89 | 稍作优化后提交 |
| 合格 | 70-79 | 需要改进后提交 |
| 待改进 | 60-69 | 需要重大修改 |
| 不合格 | 0-59 | 需要重新撰写 |

## 工作流程与质量保证

### 质量保证流程

**触发时机：**
- 每次商业分析报告生成完成后自动触发
- 与 `GENERATE.md` 配合使用

**工作流程：**
```
┌─────────────────────────────────────┐
│  步骤1: 生成商业报告 (GENERATE.md)   │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  步骤2: 自动质量审查 (本文件标准)     │
│  - 结构完整性检查                    │
│  - 内容质量检查                      │
│  - 专业性检查                        │
│  - 实用性检查                        │
│  - 生成评分和评级                    │
└──────────────┬──────────────────────┘
               ↓
         综合评分 >= 70分？
         ├─ 是 → 继续后续工作（提交管理层等）
         └─ 否 → 自动改正（最多3次）
                  ↓
        重新生成商业报告
                  ↓
         重新质量审查
                  ↓
         改正次数 < 3次？
         ├─ 是 → 继续审查
         └─ 否 → 强制通过（使用第3次版本）
                    ↓
              记录质量日志
                    ↓
              继续后续工作
```

### 改正机制

**改正触发条件：**
- 综合评分 < 70分（合格线）
- 或存在"必须修改"级别的问题

**改正次数限制：**
- 最多3次自动改正
- 每次改正后重新评分
- 第3次改正后无论评分如何，强制通过

**改正内容：**
1. **必须修改问题**（影响决策）：优先修正
   - 核心结论不明确
   - 创新点提炼不准确
   - 缺少量化数据
   - 专利布局缺失
   - FTO风险分析缺失

2. **建议优化内容**（提升质量）：时间允许则修正
   - 添加对比表格
   - 完善市场数据
   - 优化语言表达

### 与 GENERATE.md 的配合

**职责分工：**
- `GENERATE.md`：定义"如何生成"商业分析报告
- `REVIEW.md`：定义"如何检查质量"和"如何改正"
- 两者配合形成完整的质量保证体系

**调用示例：**
```
1. 读取 GENERATE.md 中的生成步骤
2. 按照步骤生成商业报告初稿
3. 读取 REVIEW.md 中的审查标准
4. 按照标准审查商业报告
5. 根据审查结果决定是否需要改正
6. 重复步骤2-5，最多3次
7. 质量合格或达到3次上限后，输出最终版本
```

### 质量阈值配置

**可配置参数：**
```yaml
quality_threshold:
  # 合格线分数
  passing_score: 70

  # 最大改正次数
  max_revisions: 3

  # 优秀线分数（可快速通过）
  excellent_score: 90

  # 强制通过后的处理
  force_pass_action: "log_and_continue"
```

### 与技术方案审查的协同

**完整文档生成流程：**
```
创意输入
  ↓
生成技术方案 (tech-solution/GENERATE.md)
  ↓
审查技术方案 (tech-solution/REVIEW.md)
  ↓
合格？
  ├─ 否 → 改正技术方案（最多3次）
  └─ 是 → 生成商业报告 (business-analysis/GENERATE.md)
           ↓
        审查商业报告 (business-analysis/REVIEW.md)
           ↓
        合格？
          ├─ 否 → 改正商业报告（最多3次）
          └─ 是 → 输出最终文档包
```

**依赖关系检查：**
- 商业报告必须依赖具体的技术方案
- 技术方案的BOM成本 → 商业报告的定价策略
- 技术方案的创新点 → 商业报告的竞争优势
- 技术方案的风险 → 商业报告的风险评估
